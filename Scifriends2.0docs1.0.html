Introduction

The web panel is programmed in PHP for the interactive web pages and uses MySQL for the database that keeps track of all data.

This document describes the functions the Web Panel database will have. There are three kinds of users: the Administrator, Researcher and Participants. Consequently, there are three different access pages, one for each of these types of users. From those access pages the users can proceed to further pages dedicated to specific functions.

 

There are three kinds of users: the Administrator, Researcher and Participants. All of them access the panel through the same link: http://abcwebpanel.mpib-berlin.mpg.de/ABCWebPanel/loginuser.php

0) how users initially sign up - questionnaire etc.

 

1) The researcher

The work done in the web panel is organized around projects. A project maintains the data necessary to select and invite participants with emails. Projects can be created. After creation the project is in the design phase where the researcher can add a link to a web survey or a web experiment (study). Note that the data collection is not done within the web panel. After filling in the details of the project the researcher can make a selection of possible participants. The selection can be made using standard demographics given at registration by all participants or from data derived from previous studies and uploaded to the database.

 

When a project is started the researcher first has to invite a random sample from the selection. Participants can be invited later from those in the selection that not yet been invited. This way the researchers can control how many participants that finishes the project and most importantly how many has to be paid.

 

Invitations are sent out with the help of email templates. Every project starts out with some basic templates for invitation and reminders, but they can be edited and new types of templates can be created. Each template can be provided in different language versions. The templates uses special code strings that are replaced with data from the database that makes the emails personalized.

 

The researcher can finish a project at any time. After finishing the page redirecting participants to the survey/experiment will display a message that it is too late to participate.

 

After collecting the data the researcher add result fields for the project. First a list of field names and there data types are defined and then an empty table is downloaded with columns for each new field, where the leftmost column is prefilled with the user ids of the participants who was invited or finished the project. This file can then be filled in by the researcher using for example excel and then uploaded to the web panel database. The uploaded columns can then be used for selecting participants in future studies.

Security of the ABCWebPanel

Overview and Authentication of Users

The ABCWebPanel follows a typical design pattern for web applications. Sensitive data is stored in a database (MySql). The web application is written in scripts using the programming language PHP. The PHP scripts generate web pages in HTML that users can access from the web. The solid line between the PHP and HTML in the diagram shows that sensitive information in the PHP scripts must be translated into HTML in order to be accessible for those who can access the pages in HTML.

Each user has to login with a valid email address and a password. The passwords are stored encrypted with SHA1 in the database. Passwords are only known by the user that provided it. To prevent unauthorized access, users have to login in to the web server. If the user provides a valid password for the email address, a PHP-session is started. Basically this means that a long random string of letters is stored in a cookie in the browser of the user. Users that do not have a valid cookie cannot access any pages on the web server. This is the technique as the one used for example in Facebook and other popular websites based on PHP. The “AGB” of the ABCWebPanel must require the users to enable cookies in their webbrowser. Otherwise the users will never be able to log in.

The ABCWebPanel has three different groups of users: participants, researchers, and the administrator. PHP-scripts for participants for example require that the user is logged in as a participant before generating any HTML. If the user is not logged in or logged as something else there will only by a short error message and a link to the login page. From the user’s point of view the ABCWebPanel is three different non-overlapping collections of web pages. In other words each PHP script only allows one kind of user to access it.

The database is also protected by usernames and passwords. When the PHP-scripts access the database a special username and password is used for each transaction. Even if someone has direct internet access to the database no data can be retrieved without the password. In the picture the database is running on a machine that is only connected to the web server and not to the internet. It is not necessary to run the database and web server on separate machines in this way but it increases security.

Information flow for the PHP Scripts

The PHP scripts are carefully written to only provide information necessary for the user and nothing else. This section gives an overview of what the different users can see and do with the data in the database.

It is also important how users are given access to the system as explained below.

 

The database administrator

The people in control of the physical machine where the database is stored can log in directly to the MySql database and make arbitrary changes to the data stored. This is important if the database get corrupted because of software errors in the PHP scripts for example.

In addition a person should be assigned a username and a password  to access only the ABCWebPanel database. This database administrator should manage regular backups of the database. Before the PHP scripts are made public to the internet, the database administrator should add a researcher called ‘admin’, which can be done without logging in as long as there is no ‘admin’ researcher. The user ‘admin’ will then be the only user that can log in as an administrator using the specified emailadress as a username.

Administrator

The administrator has access to all data in the database necessary for administrative purposes. However, this is moderated by the user interface in the web application. The administrator cannot make arbitrary changes to the database for example. For increased security the database administrator can define a set of IP-addresses that restricts which computers the administrator can log in from.

The administrator can create user accounts for researchers. In addition, if users are not allowed to freely sign up from the internet, then the administrator will create accounts for participants as well. The administrator has the right to block other users from logging in to the ABCWebPanel.

The administrator also set quotas for how much a single researcher may pay participants for participating in studies.

Researchers

Personally identifying data, such as names, email addresses or other contact information are not shown to researchers. Demographic data such as birth year, level of education and sex is used to select participants for studies. Individuals are identified by random identifiers that are uniquely assigned for each study. Thus researchers can update the system with data from the study for the purpose of selecting participants for later studies, without knowing the exact identity of the participants.

Researchers will be able to send emails to participants participating in studies. But in doing so they will not see any email addresses, unless the participants chooses to send an email to the responsible person of the study.

Researchers can share projects with other researchers, and the creator of a project can also later remove other researchers from the project.

Participants

The PHP scripts for the participants show only personal information about the participant, and what studies the participants has been invited too. There is no communication or transfer of information between participants.

Normally users sign up for the ABCWebPanel by filling in a form. Everyone who has the link to the sign up page can do so. If sign up is not open to everyone, the administrator has to gain access of emailaddresses elsewhere and invite participants via email. If the user decides to not participate in future studies the user can inactivate the account. The user can still log in and reactivate the account. The user account is not deleted because researcher may later want to collect demographic data over users that participated in a study. The ‘AGB’ showed at sign up is also always accessible from the user account.

Connecting participants to external studies

The ABCWebPanel handles participant data and information about studies, but the actual web experiment or web survey that the researcher collects data from are always external to the ABCWebPanel itself. The ABCWebPanel only provides a link such as the following example:

 

http://localhost/ABC_Web_Panel/Test/Page1.php?id=68yo4U37R27weSRF&studyid=BeXny4zmq5mFXle0

 

Codes are used to identify the participant and the study that provided the link. There is also the possibility to attach a code for the assignment to different conditions.

 

The participant has to login and start a session (a cookie is stored) before following the link to the study. The study can then link back to the ABCWebPanel when the participant has reached the end. Since the cookie is still active the participant can now be registered as having finished the study, which is stored in the database. Later the administrator can use this information to generate lists for paying the participants.

 

The experimental data is collected locally for the external studies. The security issues for the data collection depend on what kind of technology is used for the particular study. But since the ABCWebPanel provides a coded identifier for each participant, data can be collected anonymously and no additional identifying data has to be collected.

PayPal Mass Payment security for the ABC Web Panel

 

To pay using PayPal Mass Payment a username, a password and a signature has to be stored in the Web Panel database.

 

To minimize that anyone with access to machine running the database steals these codes and use them to transfer money from the web panel PayPal account, the password and signature are encrypted using AES (http://en.wikipedia.org/wiki/Advanced_Encryption_Standard), which is the standard encryption method today and considered completely safe for any purpose as long as the key is kept safe. This also make backup of the database less sensitive.

 

The administrator has a key (similar to a password) that can decrypt the password and signature at the moment of payment. As long as the key is kept secret, nobody can extract the password or the signature from our database even if they have full access to the system and all other passwords.

 

After entering and encrypting the password and the signature, the password and signature are never shown fully again. The key holder can test the key and check the first two and the last two letters of the password and the signature to see if the key is correct. Otherwise the password and signature are only used at the precise moment where a Mass Payment is done to Paypal which uses the https protocol that is standard for bank transactions for example and are never shown fully again.

 

If the system is compromised somehow a new password and signature can be generated at PayPal.

ADD QUOTA

Before a study can actually be started and participants can be invited, the study must be assigned quota for paying the participants. Also, ask Mirta to apply for money.

 

Quota

PayPal charges a small fee from each participant for receiving payment, so this must be considered when calculating the quota. Current (January 2012) fees are 1.9% + 0.35 euros. Hence, the amount entered here should be: quota = (X + 0.35) / (1 – 0.019) * sample size

In a comment, describe how the quota was calculated (e.g. [3.0 Euro + 0.41 Fees] x 65 participants). Money can be subtracted using negative amounts. PayPal fees can change with time, the latest info is on PayPal pages, e.g. :

https://www.paypal-deutschland.de/privatkunden/was-ist-paypal/gebuehren.html

 

Applying for Money

The total amount of money that needs to applied for needs to contain an additional 2% (PayPal mass payment fees) + a couple of euros (not more than 10%) buffer. There is an excel sheet prepared for calculating the quota and the amount to apply for. Amount = quota / 0.98 + buffer.

Once all is calculated, pass the following information on to Mirta: amount to apply for, name of researcher, name of study, number of participants, and basic payment for participant, length of study, and planned starting date. Once we have applied for money, the study can start.

Don't forget to inform them that the may start anew - either by sending a reminder (from the researcher's account) or sending an email from the administrator's email account (see point 5. Researcher Instructions

The ABC Web Panel is a database of people willing to participate in the studies conducted at the Max Planck Institute for Human Development. The database is connected with a web application that enables researchers to select participants according to criteria such as demographics or participation in previous studies, to send them email invitations and reminders, to start and track progress of their studies, and to process their payments. Participants are paid via PayPal.

The Panel is ideal for conducting short studies with participants in their homes instead of inviting them to the lab. This application is not designed to actually program web experiments and questionnaires. One particularly versatile and affordable solution for programming web studies is Unipark at http://www.unipark.de. There is a web version of Mouselab at http://www.mouselabweb.org. Many other tools are described at http://www.websm.org/c/1283/Software.

The ABC Web Panel has been approved by the LK, the Data Protection Committee, and the Ethics Committee of the MPIB. Participants’ anonymity is ensured by means of random identifiers that are uniquely assigned for each study. Researchers can never see participants’ personally identifying data, such as names, email addresses or other contact information. In the panel, participants are uniquely identified by their PayPal IDs, thus preventing multiple logins of same persons.

Participants and researchers can access the ABC Web Panel here: http://abcwebpanel.mpib-berlin.mpg.de. To open a researcher account please send your full name, department, email address, and if you are a student the name of your supervisor at the MPIB, to the ABC Web Panel administrator Mona Merkert at abcwebpanel@mpib-berlin.mpg.de.

The ABC Web Panel was developed by Magnus Persson, Mona Merkert, Mirta Galesic, Lael Schooler, and Uwe Czienskowski, all at the Center for Adaptive Behavior and Cognition, Max Planck Institute for Human Development, Berlin.

 

Participants

As of November 2012, more than 700 participants have joined the Panel. Most (97%) come from different areas of Germany; 66% are women; 55% have high-school or lower education; 51% are employed, 27% are students, and 14% retired; 38% are up to 30 years of age, 36% between 31 and 40 years of age, and 26% from 41 to 80 years of age. Link Method:

Direct:

[This option is normally disabled and thus not visible] Choose this if you just want to send invitation emails to participants but you do not require them to log in our system first. Choose this if you want to send invitation emails to participants and you do require them to log in our system first. In this way participants will be registered and you will be able to pay them. LR. Link with named parameters:

With this option parameters are transferred in the link sending the participant to the study. For example you entered

http://www.economicpsychology.ch/ABCWPTest/ABC_Web_Panel/Test/Page1.php

as the link to the study. The webpanel will for this option add informative parameters to your link using the parameter names 'id' for the random id code for the participant in this study, 'studyid' is the code for the project. This means that you can run several projects using the same link for study, since the studyid identifies from which project the participant came. The final link may look something like this

http://www.economicpsychology.ch/ABCWPTest/ABC_Web_Panel/Test/Page1.php?studyid=ZY60RQmVPC34SctE&id=fhH8sjSX&param1=GroupA&param2=GroupB

Responsible Person: This should be the leading researcher on the project.

Desired Sample Size: The number of participants you wish to finish your study.

Desired Number of Invitations: The number of participants you wish to invite. As not all participants will finish the study, the Number of Invitations should typically be a bit larger than the Desired Sample Size. The maximum ratio of Number of Invitations and Desired Sample Size is 3. If you exceed this ratio, a warning will appear on the Project Management page.

Maximum / Fixed Payout: The amount of money you wish to pay to each participant who finished the study appropriately. The minimum payment is 50 cents per participant. When the Payout Size is fixed, this is the amount each participant gets. When the Payout Size is variable, this is the maximum payout a participant can get.

IMPORTANT: This amount needs to include PayPal? fees. PayPal? will deduct a portion of this amount from the amount the participant gets. Therefore, this amount needs to include PayPal? fees. Current (January 2012) fees are 1.9% + 0.35 euros. Hence, the amount entered here should be = (X+0.35)/(1-0.019), where X is the actual amount you want to pay to the participants. E.g. if participants receive 1 euro, this amount will be 1.38.

Glossary

  • Participants: Members of the ABC Web Panel who participate in studies.
  • Researchers: MPIB researchers
  • Administrator: Person(s) responsible for maintaining the ABC Web Panel, inputting new researchers, assigning quotas, and executing payments.
  • Quota: Amount of money assigned to the researcher to collect data for a specific project. Desired sample size x Payout must be equal to * Quota.
  • Invitation: An email which contains an invitation to participants to log in the ABC Web Panel and complete the study.
  • Selected participants: Participants that were selected to participate in the study. The number of selected participants must be >= number of invited participants.
  • Invited participants: Those among the selected participants to whom invitation to the study has been sent.
  • Finished participants: Participants who came to the last page of the study. They may or may not have completed all items in the study – i.e., they might have skipped some items. It is recommended that the researchers implement controls within their study which prevents participants from skipping the items.

 

PayPal Mass Payment security for the ABC Web Panel

To pay using PayPal Mass Payment a username, a password and a signature has to be stored in the Web Panel database.

To minimize that anyone with access to machine running the database steals these codes and use them to transfer money from the web panel PayPal account, the password and signature are encrypted using AES (http://en.wikipedia.org/wiki/Advanced_Encryption_Standard), which is the standard encryption method today and considered completely safe for any purpose as long as the key is kept safe. This also make backup of the database less sensitive.

The administrator has a key (similar to a password) that can decrypt the password and signature at the moment of payment. As long as the key is kept secret, nobody can extract the password or the signature from our database even if they have full access to the system and all other passwords.

After entering and encrypting the password and the signature, the password and signature are never shown fully again. The key holder can test the key and check the first two and the last two letters of the password and the signature to see if the key is correct. Otherwise the password and signature are only used at the precise moment where a Mass Payment is done to Paypal which uses the https protocol that is standard for bank transactions for example and are never shown fully again.

If the system is compromised somehow a new password and signature can be generated at PayPal.
